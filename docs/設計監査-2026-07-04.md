# 設計監査 2026-07-04

監査者: Fable 5（本部 HANDOVER「Fable 残期間の高レバレッジ利用①」）
対象: recollage リポジトリ全体（実装約1,040行 / docs約780行 / ADR 10本）

## 総評: 設計（ADR）の質は高い。実装がADRとほぼ接続されておらず、「モックの意図的な省略」と「単なる未実装」の区別が記録されていないことが最大のリスク

ADR 10本は選択肢・却下理由・再評価条件まで書かれており、個人開発の設計文書として上位の質。データモデル（title null 許容・density 導出・resurfacing 用カラムの先行確保）も一貫している。

一方、フェーズ3のモックフィード実装は ADR-001/002/003/005/007/008 のいずれにも従っていない。モック段階の意図的な省略なら正当だが、**どれが「あとで払う前提の負債」でどれが「実装漏れ」なのかがどこにも書かれていない**。このままフェーズ4に入ると、モックの構造（JSON直import・モノリスページ・生hex）が既成事実として本実装に流れ込む。

## 指摘

### 高-1: プロダクトの核（ADR-008）が乱択で代替され、核仮説を検証できない

`app/composables/useFeed.ts:107-114` の `recompose()` は Fisher-Yates の全体シャッフルのみ。ADR-008 が規定するスコア式（Recency 指数減衰・Resurface 対数正規の山・間隔反復係数・セッションシードε）は一切実装されていない。`arranged`（`useFeed.ts:95-105`）も density で feature/sides/rest に振り分けるだけ。

このプロダクトの差別化本体は「出会い直し」のフィード編成であり、レイアウトではない。**現状のモックで検証できるのは見た目だけで、核体験はただの乱択**。モックを触って「良い/悪い」を判断すると、乱択の体感を核アルゴリズムの評価と誤認する危険がある。

しかも ADR-008 自身が解を用意している: `shared/utils/feedScore.ts` に純粋関数で集約し Vitest で回帰テスト、という規定。**UI に一切依存せず単体で書けて単体で検証できる部品**なので、フェーズ4 を待つ理由がない。次の一手として最も費用対効果が高い。

### 高-2: repository 層（ADR-001）不在。「差し替え1点」の約束が現状成立しない

`useFeed.ts:3-4` が `~~/data/mock/*.json` を直 import。ADR-001 の中核（UI 層はデータソースを知らない、Supabase 移行は repository 差し替えのみ）が構造として存在しない。フェーズ4 で useFeed 自体の改修が必要になる。

モック段階の割り切りとしては理解できるが、interface だけ切って `MockEntryRepository` を1枚挟むコストは小さく、ADR の約束を今から守る方が安い。

### 中-1: 密度導出が2実装・2定義（ADR-002 違反、実バグ予備軍）

- `useFeed.ts:37-41`: 大=`visual.type==='image'` / 小=`body.length<250` / 中=その他
- `scripts/import-obsidian.ts:126-130`: 大=`visual` あり / 中=`title && keyPoints.length>0` / 小=それ未満

同じ「密度」概念の判定基準が食い違う（mermaid の扱い・keyPoints 基準の有無）。ADR-002 は「shared/utils の1関数に集約」と規定済み。mermaid ビジュアルのエントリを作った瞬間、生成側と表示側で密度がズレる。`shared/utils/deriveDensity.ts` への集約を推奨（feedScore.ts と同じ置き場）。

### 中-2: clone しても動かない（data/sample/ 未作成）

`data/mock/` は gitignore 対象（著作権物を含むため正しい）だが、useFeed が直 import しているため、**clone 直後・CI 環境ではビルド自体が通らない**。コミット用ダミー `data/sample/` の作成（計画済み・未着手）か、repository 層導入時にフォールバックを持たせるかのどちらかが必要。CI を導入する（後述）前提条件でもある。

### 中-3: フェーズ3成果物が丸ごと未コミット、STATUS.md が実態と逆

git は init 1コミットのみで、`app/pages/`・`app/composables/`・フィード実装は全て untracked。作業喪失リスクがある。また本部側 STATUS.md（2026-06-14）は「フェーズ3未着手・リモート未作成」のままで、実態（リモートあり・フェーズ3部分実装済み）と逆。次の事業部セッションで①フェーズ3をコミット②STATUS 更新、をセットで。

### 中-4: ADR-003 のレンダリング方針が nuxt.config に未反映

ADR-003 は「デフォルト CSR（`ssr: false` 相当）＋ routeRules ハイブリッド」を採用したが、`nuxt.config.ts` に routeRules も ssr 設定もなく Nuxt デフォルト（SSR 有効）のまま。現状モックでは実害が出ていないが、決定と設定ファイルが不一致の状態。

### 低（モック負債として明記すれば許容）

- `app/pages/index.vue` 536行モノリス（ADR-005 の2層コンポーネント未実装、scoped CSS 347行込み）
- 生 hex 直書き全面（`index.vue:190-536`）。ADR-007 の Primitive/Semantic トークン・`@theme` 未導入。Claude Design からの「値の逆抽出」が未実施なので順序としては自然だが、逆抽出タスクが open-questions に留まったまま
- テスト0・CI0（architecture.md は「必須装備」と明記）。feedScore.ts 着手と同時に Vitest だけ先に入れるのが低コスト
- 自作 YAML パーサー（`import-obsidian.ts:43-77`）の脆さ。既に Audible タイトル誤抽出の実バグ実績あり。個人 Vault 前提の割り切りは可だが、誤パース時に**無音で壊れる**のが問題なので、想定外構造の検知時に throw する防衛だけ足す価値あり
- `useFeed.ts:88` のコメント（new Date を避ける）と実装（曜日算出で new Date 使用）のねじれ。決定論的入力なので実害なし、コメント修正のみ

## 推奨対応（優先順）

1. **フェーズ3のコミット＋STATUS.md 更新**（作業喪失リスクの解消が先。監査指摘全部より優先）
2. **feedScore.ts の実装＋Vitest 導入**（UI 不要・純関数・ADR-008 の規定通り。核仮説の部品が固まり、モックに配線すれば「出会い直し」の体感検証が初めて可能になる）
3. **deriveDensity の shared/utils 集約**（2定義の統一。feedScore と同時にやると安い）
4. **repository interface の導入＋data/sample/ 作成**（フェーズ4 の前提整備。clone 可能性も回復）
5. **モック負債リスト**を docs に1枚（index.vue モノリス・生hex・SSR設定など「意図的にあとで払う」項目を列挙し、フェーズ4 の入り口でチェックリスト化する）

設計そのもの（ADR の判断内容）に覆すべきものは見つからなかった。監査の結論は「設計を直せ」ではなく「**設計は良いので、実装を設計に追いつかせる順序を決めろ**」。
