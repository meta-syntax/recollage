## タスク一覧（依存順）

### 1. 詳細ページ — フェーズ3.5（次の実装候補）

「つづきを読む」「カードクリック」の遷移先。現状は導線が死んでいる。

- **検証仮説の本丸**: 「読み返すか」はフィードを眺めることではなく詳細を開くことで初めて成立する
- **Resurface 検証の前提**: `lastViewedAt` は詳細を開いたときのみ更新（ADR-008）。詳細ページが無い限り viewCount 項は永遠に検証できない（mock-debt #4）
- 設計論点: ルート設計（`/entries/[id]`）、本文Markdownレンダリング、閲覧記録の書き込み先（モック段階は localStorage か、フェーズ4を待つか）
- 依存: なし（repository に `getEntry(id)` を足すだけで着手可能）

### 2. CMS・入力フォーム — フェーズ4の本丸

「＋ 記録する」ボタンの遷移先。既製CMS不採用（ADR-006）＝自作。

- **着手前に設計セッションが必要**（open-questions 優先度高「入力・編集体験の設計」）: 書き捨てキャプチャの摩擦最小化、保存後の構造化促し、編集導線
- 依存: 書き込み系 repository（＝Supabase 移行・フェーズ4）とセット。フェーズ3の手応え待ち

### 3. カテゴリ追加・管理画面 — フェーズ4（CMSの一部）

- 設計論点: カテゴリツリーの D&D 管理、sortOrder の順序表現（open-questions 学習待ち「fractional indexing」）
- 依存: 2 と同じく書き込み系。入力フォームより後でよい（現状カテゴリはインポータの固定ツリー）

### 4. MCP サーバー提供 — 超拡張（フェーズ5+のアイデア置き場）

AI と壁打ちした結果を、MCP 経由で Recollage にエントリとして書き込む。

- 構想: Recollage が MCP サーバーを公開し、壁打ちセッションの学び・結論を `create_entry` ツールで直接記録 → フィードの「出会い直し」対象になる
- 効いてくる資産: repository 抽象（ADR-001）と API レイヤー折衷（ADR-004）が入り口を規定済み。書き捨てエントリへの LLM 自動補完（open-questions 先送り）と相性が良い
- 依存: 書き込み系 API（フェーズ4完了後）。それまでアイデアとして凍結
