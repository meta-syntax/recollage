# ADR-003: レンダリング戦略 — routeRulesハイブリッド、デフォルトCSR

- ステータス: 採用
- 日付: 2026-06-11

## コンテキスト

SSRが（特に公開コンテンツ面で）標準的になっている潮流があり、レンダリング戦略を明示的に決める必要がある。SSRの利点は「初回訪問者にJSロードを待たせずHTMLを届ける」ことで、SEO・匿名初訪・コンテンツ配信で効く。

Recollageの特性はその逆を向いている:

- 認証前提の単一ユーザーアプリ。検索エンジンに見せるページが存在しない
- リピート訪問が主。初回ロードより再訪時のキャッシュ効率が効く
- 可変マソンリーはクライアントでのJS測定が必要で、サーバがHTMLを先に返してもレイアウト確定はクライアント待ち
- SSRはハイドレーション制約（window非依存・サーバ/クライアント出力一致）の税を全コンポーネントに課す

また認証付きフィードのSSRは、サーバ側でセッション検証→データ取得→レンダリングが終わるまで1バイトも返らない（TTFB悪化）ため、銀の弾丸ではない。

## 決定

routeRulesでルート単位にレンダリングモードを宣言する。デフォルトはCSR。

```typescript
routeRules: {
  '/**': { ssr: false },
  // 将来、エントリの公開共有面を作ったら該当ルートのみ prerender / ssr を有効化
}
```

## 既知の弱点と対策

CSRの構造的弱点: 初期ロードで「JSダウンロード→実行→データ取得→描画」が直列に走る。以下を採用する:

1. **スケルトンUI**: spa-loading-template にフィードのスケルトンを置き、白画面を排除する
2. **前回フィードのローカルキャッシュ**: IndexedDBに前回データを保存し、起動時は即描画→裏で最新に差し替え（クライアント側 stale-while-revalidate）
3. **Service Worker / PWA**: アセットのプリキャッシュで再訪時のネットワーク待ちを排除。プッシュ通知要件でSWはどのみち導入する
4. **バンドル規律**: Mermaidライブラリは必ず遅延ロード。バンドルサイズ予算をCIでチェック
5. **画像最適化**: カード用サムネイル生成・遅延読み込み（ビジュアル中心フィードのLCPの本丸はJSより画像）

## 再評価条件

- エントリの公開共有機能を作るとき → 該当ルートのみ prerender / ssr 化
- 上記対策を入れてもコールド初回ロード（新端末・キャッシュ消去後）が体感問題になったとき → フィードルートのみSSR化を検討。repository抽象（ADR-001）によりデータ層は無傷で切り替え可能
