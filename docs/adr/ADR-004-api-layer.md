# ADR-004: APIレイヤー — Supabase直＋秘匿系のみNitro

- ステータス: 採用
- 日付: 2026-06-11

## コンテキスト

データアクセスの経路として3案を比較した。

- A. クライアント→Supabase直（RLS保護）: Supabaseの標準形だが、LLM・画像生成のAPIキーをクライアントに置けないため単独では成立しない
- B. 全て Nitro（server/api/）経由: 統制は最強だが、単一ユーザーアプリのCRUDのために全APIを自前で書くのは過剰投資
- C. 折衷: CRUDはSupabase直、秘匿キーが要る処理のみ server/api/

## 決定

**C（折衷）** を採用する。

- CRUD・検索: クライアントからSupabaseを直接呼ぶ（RLSで保護）
- 画像生成・LLM呼び出し・embedding生成: `server/api/` に置く（APIキー秘匿）
- 前提として、クライアント側はrepository層（ADR-001）で抽象化されており、呼び先の違いはrepository実装に閉じる

## 帰結

- サーバコードは秘匿が必要な最小限に保たれ、CRUDの二重実装（API定義＋クライアント）を避けられる
- UI層は「Supabase直か `/api/` 経由か」を知らない。経路の変更はrepository実装の修正のみ

## 再評価条件

- マルチユーザー化、またはサーバ側に置くべきビジネスロジックが肥大したとき → B（全面Nitro経由）への移行を検討
