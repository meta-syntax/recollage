# ADR-008: フィード編成アルゴリズム

- ステータス: 採用
- 日付: 2026-06-14

## コンテキスト

フィードはこのアプリの主戦場であり、「新着＋忘れかけ＋関連の編集的混合」「開くたびに誌面が変わる」「スクロール中は並びが変わらない」という要件（構想ログ／open-questions 優先度高）を満たす編成ロジックが必要。最も難しく、最も独自性が出る部分。

制約条件:

- **モック段階（フェーズ3）は embedding を持たない**（data-model.md）。本来の意味の「関連」は計算できない
- **インポートデータは `lastViewedAt` が全件 null**（一度も開いていない）。素直に resurface を実装すると初回フィードがただの新着順になり、検証（スクロールして読み返すか）が成立しない
- 編成ロジックは `useFeed` composable に置き、repository（ADR-001）の裏を知らない。スコア計算の本体は `shared/utils/` の純粋関数に切り出し、Vitest でテストする

## 決定

### スコア式

各エントリ `e` にスコアを与え、降順に並べる。

```
S(e) = w_new · Recency(e) + w_resurface · Resurface(e) + w_related · Affinity(e) + ε(seed, e)
```

各項は 0〜1 に正規化。重みはチューニング前提で設定値として外出しする（初期値 `w_new = 0.5`, `w_resurface = 0.5`, `w_related = 0`）。

### 1. Recency（新着）

```
Recency(e) = exp(-Δcreated / τ)      Δcreated = now - createdAt
```

- 作成からの経過を指数減衰。`τ` は数日オーダー（初期値 7日）。新規エントリの初速を与える。

### 2. Resurface（忘れかけ）— 間隔反復の山型

**時間基準は `lastViewedAt ?? createdAt`**。未閲覧なら「書いた日が最後に触れた日」とみなす。これにより初回（コールドスタート直後・全件 null）から「昔書いて忘れていた学びが浮上する」挙動が成立する。

最適間隔は閲覧回数で伸びる（定着するほど思い出す頻度を下げてよい＝間隔反復 / SM-2 的）。

```
Δseen  = now - (lastViewedAt ?? createdAt)
I(e)   = I_base · factor^viewCount          // 初期値 I_base = 7日, factor = 2
Resurface(e) = exp( -( ln(Δseen / I(e)) )² / (2σ²) )   // 対数正規の山型, σ 初期値 0.6
```

- **直後は低い**（さっき見た）→ **最適間隔付近でピーク**（そろそろ忘れる頃）→ **放置しすぎると緩やかに下がる**。対数スケールのガウシアンでこの3挙動を1式で表す。
- `viewCount` が増えると山のピーク位置が右（より長い間隔）へ移動する。

### 3. Affinity（関連）

**フェーズ3では `w_related = 0`**（embedding 不在のため項を無効化）。式の形だけ最初から持ち、フェーズ4で pgvector のコサイン類似度を `Affinity(e)` に接続して重みを上げるだけにする。フィードのトップはコンテキストを持たないため、関連は「直近開いたエントリ群への近さ」をシードに計算する想定（詳細はフェーズ4で別ADR）。

### 4. ε（誌面の揺らぎ）— セッション固定

```
ε(seed, e) = (hash(seed, e.id) を [-a, +a] に写像)     // a 初期値 0.15
```

- `seed` はアプリ起動時に1回だけ生成し、セッション中は不変。これにより:
  - **開くたびに誌面が変わる**（起動ごとに seed が変わる）
  - **スクロール中は並びが変わらない**（同一セッション内は同じ seed → 同じ順序、決定論的）
- 振幅 `a` は上位群を軽くシャッフルする程度に抑え、スコアの大小関係を破壊しない。

### 露出の減衰

`lastViewedAt` は **詳細を開いたときのみ** 更新する（フィードに表示されただけでは更新しない）。見られなかった忘れかけは浮上し続ける＝正しい挙動。「同じ忘れかけが毎回最上位に居座る」問題は、(a) Resurface の山型がピークを過ぎると自然に下降すること、(b) セッション seed の揺らぎ、の2つで緩和する。専用の露出ペナルティ状態は持たない（保存コストを避ける）。

### マソンリーとの関係

スコアは「フィードに載せる順序」を決める。可変マソンリーの高さ詰め（ADR-002 の密度別カード）はレイアウト層の責務で、スコア順を視覚的な厳密順序に固定はしない。重要なものが上方に集まれば足りる。

## 帰結

- スコア本体は `shared/utils/feedScore.ts`（純粋関数）に集約。重み・τ・I_base・factor・σ・a は設定オブジェクトで注入し、検証中にチューニングできる。Vitest でカーブ特性（直後低・最適間隔ピーク・過放置で下降）を回帰テストする
- フェーズ3は2軸（Recency + Resurface）＋揺らぎで検証する。3軸目（Affinity）は無効化したまま形だけ通す
- フィード編成の未決（open-questions 優先度高の片方、および「resurfacing の具体アルゴリズム」）はこのADRで解消。残るゲートは「入力・編集UX」のみで、これはフェーズ4の領域でありフェーズ3（モックフィード検証）には不要

## 再評価条件

- embedding 導入（フェーズ4・Supabase + pgvector）時 → Affinity を実装し `w_related` を有効化、別ADRで関連シードの決め方を確定
- 検証で「同カテゴリが固まりすぎる／散りすぎる」が問題になったら → カテゴリ多様性（MMR 的間引き）または逆に特集クラスタリングを追加検討
- 重みチューニングだけで足りる調整は ADR を起こさず設定値で対応する
