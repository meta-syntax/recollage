# ADR-007: CSS戦略 — Tailwind v4＋デザイントークン2層

- ステータス: 採用
- 日付: 2026-06-11

## コンテキスト

「雑誌の見た目」が差別化の本体であるため、CSS戦略は場当たりにできない。選択肢はTailwind系・ヘッドレスUIライブラリ・完全自作。

現行のTailwind v4は設定がCSSファーストで、`@theme` ディレクティブによりデザイントークンをCSS変数として宣言し、そこからユーティリティクラスが生成される。「デザイントークンを渡す」運用はv4ではオプションではなくアーキテクチャそのもの。

## 決定

1. **Tailwind v4 を採用**する
2. **トークンは2層構造**
   - Primitive層: 生の値（カラーパレット・タイプスケール・スペーシングの目盛り）
   - Semantic層: 用途の名前（`surface-card`、`text-muted`、`density-lg-gap` 等）
3. **feature層コンポーネントはsemanticトークン由来のユーティリティのみ使用可**。primitive直書き（`gray-500` 等）が許されるのは `components/ui/` まで（ADR-005の依存ルールと対応）
4. **トークンの値は先に決めない**。デザイン探索から逆抽出する:
   1. Claude Design（Anthropicのデザイン生成ツール、2026-04リサーチプレビュー）で、アートディレクションを明示的に変えた複数案の叩き台を出す（例: エディトリアル系・ギャラリー系・新聞系）
   2. 1案を選定
   3. 色・タイプスケール・余白のリズムを逆抽出し `@theme` のトークンに落とす
   4. ui/ コンポーネントをトークンで実装

   和文タイポグラフィ（見出しフォント・行間）もこの探索の中で決める

## 帰結

- 「雑誌の顔」の変更はsemantic層の再マッピング1箇所で全体に波及する。ダークモード対応も同様
- AI生成デザインは無難で既視感のある方向に収束しがちなため、複数アートディレクションの比較を叩き台プロセスの必須手順とする（1案目で決めない）
- ヘッドレスUI部品ライブラリ（モーダル・ドロップダウン等のアクセシビリティ部品）の採否は必要が生じた時点で判断する（open-questions参照）。フィード検証フェーズには不要
