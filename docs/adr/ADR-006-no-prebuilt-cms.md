# ADR-006: 既製CMS不採用 — コンテンツ管理面は自作する

- ステータス: 採用
- 日付: 2026-06-11

## コンテキスト

エントリの入力・編集・分類を担う面（CMS面）について、ヘッドレスCMS等の既製品を使う選択肢がある。ブログ事業部ではPayload CMSの採用実績があり、既製品に乗ること自体は選択肢として現実的だった。

## 決定

既製CMS（ヘッドレス含む）は採用しない。コンテンツ管理面は自作する。

## 理由

1. **入力体験そのものが差別化の本体** — 「最低限、本文だけで保存できる（書き捨てOK）」「保存後に構造化フィールドへの入力を促す」「構造化は強制ではなくインセンティブ」という入力UXはプロダクトのコア要件であり、既製CMSの管理画面に乗せると、最も作り込むべき部分を既製品の枠に明け渡すことになる
2. **「CMS」が単一の管理画面ではなく、複数の機能ドメインに分かれる** — このアプリのコンテンツ管理は以下の異なるUX面の集合であり、既製CMSの汎用管理モデル（コレクション＋編集フォーム）に素直にマッピングできない:
   - クイックキャプチャ（書き捨て入力、摩擦最小）
   - 保存後の構造化促し（エンリッチ導線）
   - 既存エントリの編集（加筆・ビジュアル再生成）
   - カテゴリツリーのD&D管理
   - Obsidian一括インポート

   既製品に要件を合わせ込むマッピング作業の方が、要件に沿って自作するより高くつく。**結果的に自作が早い**

## 帰結

- 入力・編集UXの設計タスクが正面から発生する（`open-questions.md` 優先度高に登録）
- CRUDの土台はADR-004（Supabase直＋repository抽象）の上にそのまま乗り、追加のインフラは不要
