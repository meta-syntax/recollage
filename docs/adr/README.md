# ADR（Architecture Decision Record）運用ルール

アーキテクチャ上の意思決定を1決定1ファイルで記録する。「なぜそうしたか」を後から辿れる状態を保つのが目的。

## 新規作成

1. `TEMPLATE.md` をコピーする
2. ファイル名は `ADR-NNN-<kebab-title>.md`
   - `NNN` = 既存の最大番号 +1（ゼロ詰め3桁）
   - タイトルは英小文字ケバブケース（例: `css-strategy`）
3. ステータスは `提案中` で起票し、合意できたら `採用` に変える

## ステータスの許容値

| 値 | 意味 |
|---|---|
| 提案中 | 起票済み・未合意 |
| 採用 | 有効な決定 |
| 棄却 | 検討したが採用しなかった（記録として残す） |
| 非推奨 | かつて採用したが、もう推奨しない（置換先がない場合） |
| 置換（ADR-NNN により置換） | 別ADRに取って代わられた。本文は変えず、この行だけ書き換える |

## 理由の書き方（重要）

「理由」には**客観的に検証できる根拠だけを書く**。判断基準は「誰が読んでも同じ結論を再導出できるか」。

- 書いてよい: 技術的トレードオフ、コスト（維持・移行・追従）、制約、計測可能な事実、公式の推奨経路
- 書かない: 「好み」「足りる気がする」「なんとなく」等の主観・感情。これらが出てきたら根拠の欠落のサインなので、その選択を支える客観的事実に置き換える

主観でしか説明できない決定は、ADRにする段階ではない（まだ根拠が言語化できていない）と考える。

## 改訂のルール（重要）

**採用済みADRは不変（イミュータブル）として扱う。** 決定は採用時点のスナップショットであり、後から本文を書き足して育てない。ADRに記録するのは「何をなぜ決めたか」であって、導入手順・設定の書き方・使い方といった実装メモは書かない（それらは `../architecture.md` や README、コード自身に置く）。

変更が生じたときの対応:

- **決定が変わる / 覆る / 拡張する** → 既存ADRに追記しない。**新しいADRを起こして置換する**。旧ADRはステータスを `置換（ADR-NNN により置換）` に変え、本文はそのまま残す（過去の判断の記録として保全する）。新ADRのコンテキストに「ADR-NNN を置換」と明記する
- **誤字・リンク切れ・表現の修正** → 決定の内容を変えない範囲でその場で直す。履歴注記は不要

「決定の中身に手が入るか？」で振り分ける。入るなら新ADR。入らない字面の修正だけがその場編集の対象。

## 一覧

連番と現行ステータスは各ファイル先頭を参照。決定の全体像は `../architecture.md`、未決事項は `../open-questions.md` にある。
