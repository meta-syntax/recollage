# ADR-011: CMS入力・編集体験の設計（キャプチャ／構造化促し／編集／カテゴリ管理）

- ステータス: 採用
- 日付: 2026-07-05

## コンテキスト

既製CMS不採用（ADR-006）により、コンテンツ管理面の設計タスクが正面から発生していた（open-questions 優先度高「入力・編集体験」）。ADR-006 が列挙した機能ドメインのうち、フェーズ4で作る4面を設計セッション（2026-07-05）で確定する:

1. 書き捨てキャプチャ（摩擦最小化）
2. 保存後の構造化促し
3. 既存エントリの編集導線
4. カテゴリツリーの管理

前提・制約:

- データモデルは書き捨てを先取りしている: `body` が唯一の必須フィールド、`title: null` = 書き捨て、`categoryId: null` = 断片、表示タイトルは body 先頭行でフォールバック（shared/types/entry.ts）
- 構想の使用モデル: 「最初はラフに分類して、溜まってきたら整理する」（構想メモ）。整理は稀だが一括・大量移動になる
- 構造化は強制ではなくインセンティブ（ADR-006 の理由1）
- 書き込み系 repository（フェーズ4・Supabase）とセットで実装する。本ADRは設計のみ

## 決定

### 面1: 書き捨てキャプチャ

- 入口は誌面の「＋ 記録する」ボタン。**モーダル**で開く（ページ遷移なし）
- フォームは **body の textarea 1個**と保存のみ。title・カテゴリ・keyPoints は出さない（null で保存 → 断片・小カードとして誌面に載る）
- 保存は Cmd+Enter。「見出し＋数行」型の書き捨ては1行目に見出しを書けば、先頭行フォールバック規約でタイトル付きと同じ見た目になる（フォームを分けない）
- スマホ入口・専用キャプチャページは今はやらない

### 面2: 保存後の構造化促し

- 保存直後に**「整える」リンク付きトーストを1回**出す。押せば編集ページ（面3）へ、無視すれば消える
- 恒常的な促しは**誌面そのものをリマインダーにする**: 断片カードに未整理の徴を付け、ε・Resurface で誌面に浮上したとき目に入る。読み返す価値を感じたら整える。整えられずに流れる断片は「整理しなくていい」という判定そのもの
- 能動的な一括整理の導線は目次の「断片」リンク（断片一覧）

### 面3: 既存エントリの編集

- **専用ページ `/entries/[id]/edit`** に一枚のフォーム
- 入口は3つ、着地は同一: 保存直後のトースト／断片カードの徴／詳細ページの「編集」ボタン
- フィールド: title / body / keyPoints / background / categoryId / visual
  - keyPoints は「1行=1項目」のテキストエリア
  - visual は type の select（image / mermaid）＋ content の入力欄（image は URL、mermaid はコード）。生成・添付は先送りのまま（open-questions「ビジュアライズ手法の振り分け主体」）
  - カテゴリ選択はインデント付き select 1個（ツリー管理は面4）

### 面4: カテゴリツリーの管理

- **専用ページ `/categories`**。ツリー表示＋追加（子を生やす）／リネーム／削除
- 移動・階層変更は**D&D**（構想メモの要件を維持。「溜まってきたら整理する」一括整理の場面で D&D の価値が最大化するため）
- 順序表現は **sortOrder 整数連番のまま**。D&D 確定時に影響する兄弟全件を書き換える。単一ユーザー・1トランザクションでは全件書き換えで十分であり、fractional indexing は不採用（open-questions 学習待ち「カテゴリの順序表現」はこれで解消）
- 削除の仕様:
  - 子カテゴリを持つカテゴリは削除不可（先に子を移動/削除）
  - エントリが付いているカテゴリは、確認の上でエントリを断片（categoryId: null）へ退避
  - カスケード削除（エントリごと削除）は設けない

## 検討した代替案

- **専用キャプチャページ（`/entries/new`）**: 保存後に戻る先が誌面なので、遷移がある分だけ摩擦が増える。棄却。スマホ・PWA 要件が出たら URL 付き入口（`/capture`）を追加する
- **書き捨てフォームに任意のタイトル欄**: 先頭行フォールバック規約で「見出し＋数行」型を吸収できるため不要。棄却
- **保存直後の即モーダルで構造化を聞く**: 面1で削った摩擦を裏口から戻す。棄却
- **編集をモーダルで行う**: 編集は腰を据える行為でフォームが縦に長く、トーストから飛ぶ URL も要る。棄却
- **並べ替えを上下ボタンにする（D&D 取りやめ）**: 実装コストは低いが、使用モデル「ラフに分類→溜まったら一括整理」では移動が一括・大量になり、上下ボタンとの操作差が最大化する場面が想定の中心にある。コスト単独は要件変更の根拠にならず棄却（セッション内で提案→撤回の経緯あり）
- **fractional indexing の採用**: 必要になるのは項目数が多いか同時編集があるとき。単一ユーザーの全件書き換えで足りる現段階では過剰。棄却

## 帰結

- open-questions の「入力・編集体験（CMS面）の設計」（優先度高）と「カテゴリの順序表現」（学習待ち）が解消される
- フェーズ4の実装タスクが確定する: 書き込み系 repository メソッド（createEntry / updateEntry / カテゴリ CRUD・reorder）、キャプチャモーダル、トースト、`/entries/[id]/edit`、`/categories`
- ツリー D&D の実装コストを引き受ける。実装時の論点はライブラリ選定（Vue のツリー D&D 対応）に絞られる
- 断片カードに未整理の徴を付ける小改修がフィード側に入る

## 再評価条件

- スマホからのキャプチャが要件化したとき（`/capture` 入口・PWA の検討）
- カテゴリの項目数が大きく増えるか同時編集が要件化したとき（fractional indexing の再検討）
- 検証仮説の判定で「断片が浮上しても整えない」が常態なら、面2の設計（誌面リマインダー方式）を見直す
