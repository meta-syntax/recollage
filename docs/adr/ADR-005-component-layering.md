# ADR-005: コンポーネント2層構造とNuxt Layers不採用

- ステータス: 採用
- 日付: 2026-06-11

## コンテキスト

対外公開（ポートフォリオ）を見据え、コンポーネントの共通化・モジュール化の方針を場当たりにしたくない。Nuxtには公式のモジュール化機構（Layers）もあり、採否を明示しておく必要がある。

## 決定

`app/components/` を2層に分け、依存ルールを3つ定める。

```
components/
├── ui/        # 第1層: デザインプリミティブ（BaseCard, BaseButton, BaseTag, BaseModal ...）
├── feed/      # 第2層: フィーチャー
├── category/
└── visual/
```

1. **ui/ はドメインを知らない** — `Entry` 等のドメイン型をimportしない。propsはプリミティブ型のみ
2. **フィーチャー間の相互importは禁止** — 共有したくなったら ui/ に降ろすか composable に抽出する
3. **pages/ は組むだけ** — レイアウトとフィーチャーの組み合わせのみ。ロジックを書かない

Nuxt Layers は**不採用**。Layersは複数アプリ間の共有や大規模なドメイン分割のための機構であり、単一アプリの現段階ではディレクトリ規約＋依存ルールで十分。

## 帰結

- 「雑誌の見た目」を構成するui部品が、データ構造の変更から完全に切り離される
- フィーチャー間の暗黙の結合が構造的に防がれる。共通化は明示的な「降ろす」操作として行われる
- カードはサイズ別コンポーネント（FeedCardLarge / Medium / Small）に分割する。密度ごとに表示要素そのものが異なるため、propsで切り替える単一カードより条件分岐が減り保守的に有利

## 再評価条件

- 公開共有面など、別アプリ性のある面が育ったとき → Nuxt Layers の採用を再検討
