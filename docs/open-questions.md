## 設計が必要（優先度高）

- ~~**フィード編成アルゴリズム（useFeed の中身）**~~ → **ADR-008 で解消（2026-06-14）**。スコア式・間隔反復の山型 resurface・セッションシードによる順序安定／揺らぎを確定。フェーズ3は Recency + Resurface の2軸（Affinity は embedding 待ちで無効化）
- **入力・編集体験（CMS面）の設計** — 既製CMS不採用は確定済み（ADR-006）。残るは自作する各面の設計: 書き捨てキャプチャの摩擦最小化、保存後の構造化促し（即モーダルか・後からリマインドか）、既存エントリの編集導線、カテゴリツリーのD&D管理。機能ドメインが複数に分かれるため面ごとに設計する。**フェーズ4の領域であり、フェーズ3（モックフィード検証）には不要**

## 設計が必要（優先度中）

- **古いアーカイブでの ε 支配（コールドアーカイブ問題）** — ADR-008 実装後の実測（2026-07-05）で発見。現在の実データは全21件が作成100日以上前・未閲覧のため、Recency ≈ 0 かつ Resurface ≈ 0（対数正規の山の裾野。σ=0.6 では Δ=100日 vs I=7日 で ≈5e-5）となり、スコアが実質 ε（±0.15 の揺らぎ）のみ＝**乱択と区別がつかない**。`lastViewedAt ?? createdAt` のコールドスタート対策は「作成日が数週間以内」のときだけ効く。一方 data/sample/（直近日付・閲覧履歴あり）では最適間隔付近のエントリが特集に正しく浮上することを確認済みで、実装ではなくパラメータとデータ鮮度の問題。候補: (a) σ を広げて裾野を緩やかにする（1.2〜1.5） (b) I_base を大きくする (c) インポート時に時間基準を擬似的に散らす (d) 全項が裾野に落ちたときのフォールバック項を足す。「重みチューニングだけで足りる調整は設定値で対応」（ADR-008）の範囲か、フォールバック項の追加（ADR改訂）かの判断が要る

## 学習・調査待ち

- **カテゴリの順序表現** — sortOrder整数連番はD&D並べ替えで兄弟全件の書き換えが要る。fractional indexing 等の代替を学習してから再議論
- **lint強度の引き上げ** — ESLint方針はADR-009で確定（Stylistic一本化）だが、型認識ルール（`recommended-type-checked`）・strict系は未有効。`tsconfigPath` で型認識を、`strict: true` で strict を opt-in できる。コード量が増えて実行時バグ検出の価値が上がった段階で判断する
